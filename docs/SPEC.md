## v2.0 新增规范 (2026-03-23)

### 一、手势交互体系

所有卡片和车端界面统一遵循以下手势语义，新功能设计必须参照此表。

| 手势 | 语义 | 适用场景 | 参考规范 |
|------|------|----------|----------|
| **点击** | 主操作 / 进入详情 | 卡片→显示态、按钮触发、新闻展开详情 | Apple HIG: Tap |
| **左右滑** | 切换同层内容 | 车端卡片切换、新闻流翻页 | Material 3: Swipe |
| **上下滑** | 滚动内容 | 新闻流滚动、日程列表滚动、闹钟列表滚动 | 通用 scroll |
| **长按** | 进入编辑态 / 危险操作 | 车端→编辑态、闹钟/日程删除确认 | Apple HIG: Long Press |
| **长按拖动** | 排序 | 车端编辑态卡片排序 | Material 3: Drag |
| **左滑（列表项）** | 显示操作按钮 | 闹钟/日程列表项→滑出删除按钮 | iOS: Swipe Actions |

**禁止**：双指缩放（车载场景不适用）、hover 状态（触屏无鼠标）、小于 76×76dp 的触控目标。

**车载安全约束**：
- 驾驶中禁止需要阅读 >20 字的操作
- 主操作必须单次点击完成（不允许双击、三击）
- 操作反馈必须有视觉 + 触觉（如 haptic）确认

### 二、卡片唯一标识体系

**问题**：v1.0 同一张卡片可被重复推送到车端。

**规范**：

1. **Widget ID 生成**：`widget_id = SHA256(component_type + theme + JSON.stringify(params)).slice(0, 16)`
   - 相同类型 + 相同主题 + 相同参数 = 相同 ID
   - 修改任意参数（颜色、城市、日期等）= 新 ID

2. **推送行为**：
   - 车端已有同 `widget_id` → **更新**（覆盖旧版本，不新增）
   - 车端已有同 `component_type` 但不同 `widget_id` → **提示**"替换现有的 X 卡片？"
   - 车端无此卡片 → **新增**

3. **车端持久化**：`localStorage` 存储 `accepted_widgets: [{widget_id, component_type, params, received_at}]`，页面刷新后恢复

4. **服务端**：`POST /api/sync` 响应增加 `action: "create" | "update" | "replace"` 字段

### 三、手机端 / 车端数据分层

**原则**：手机端用精品 mock 数据保障最佳预览体验，车端用真实数据保障功能可用性。两端互不影响。

| 卡片 | 手机端（preview 模式） | 车端（live 模式） |
|------|----------------------|------------------|
| 天气 | 精品 mock（多种天气可选，标注"示例数据"） | 和风天气 API 真实数据 |
| 新闻 | 精品 mock（高质量样本，标注"示例内容"） | RSS + AI 摘要实时数据 |
| 日程 | 示例日程（3-5 条，标注"推送后显示真实日程"） | 车端系统日历 API（JSBridge） |
| 闹钟 | 示例闹钟（3 条，标注"推送后显示真实闹钟"） | 车端系统闹钟 API（JSBridge） |
| 音乐 | 精品 mock（固定封面+曲目+颜色） | 真实 MediaSession（JSBridge） |
| 纪念日 | 用户输入数据（手机端即真实数据） | 同手机端参数 |

**实现方式**：
- 模板 HTML 增加 `data-mode` 属性：`preview` 或 `live`
- 手机端 `render-widget.js` 注入 `mode=preview`
- 车端 `car-simulator.html` 注入 `mode=live`
- preview 模式显示右上角"示例数据"角标（半透明，不影响预览体验）

### 四、车端"编辑态"

**双态设计**：

**显示态**（默认）：
- 全屏渲染当前卡片
- 左右滑动切换卡片
- 底部 dot 指示器
- 点击卡片 → 进入卡片内交互（如天气切换城市、新闻展开详情）

**编辑态**（长按触发）：
- 卡片缩小至 80%，露出操作 UI
- 右上角 ✕ 删除按钮（点击弹出确认）
- 左右滑动浏览所有卡片（缩略图级别）
- 长按拖动 → 调换卡片顺序
- 点击任意卡片 → 退出编辑态，进入该卡片的显示态

**状态转换**：
```
显示态 —— 长按 ——→ 编辑态
编辑态 —— 点击卡片 ——→ 显示态
编辑态 —— 长按拖动 ——→ 排序（松手后仍在编辑态）
编辑态 —— 点击 ✕ ——→ 删除确认弹窗 ——→ 删除/取消
```

### 五、各卡片 v2.0 功能需求

#### 天气
- 城市搜索：支持输入任意城市名，调用和风天气 GeoAPI 实时搜索（不再限制为 6 个固定城市）
- 动态数据：切换城市后必须展示该城市的真实天气数据
- 微调反馈：无法执行的微调指令需返回明确提示（而非静默忽略）

#### 日程
- 颜色选择器修复：新建日程弹窗的颜色点击可选中
- 完成功能：支持 ✅ 标记日程完成，完成后的日程灰色显示或隐藏
- 添加按钮：FAB 尺寸 ≥76×76dp，增加光晕/阴影突出显示

#### 音乐播放器
- MediaSession 真实对接：车端通过 JSBridge 获取正在播放的内容、控制播放
- 颜色一致性：`primary_color` 参数在推送链路中完整传递，车端/手机端颜色一致
- 空状态设计：无播放内容时显示引导界面（对齐 Pencil 最新设计）

#### 纪念日（恋爱/放假/宝宝）
- 修复图片映射（午后咖啡/樱花小径）
- 彩蛋微动效化：粒子数 5-8，从点击位置辐射，渐隐消失（恋爱=小爱心、放假=彩带、宝宝=小鸭子积木）
- 图片上传压缩：max 600px + JPEG 0.5，超 200KB 提示用户
- 名字显示：用户输入的名字信息必须体现在卡片中
- 三卡对齐：共享统一的 glass-panel 布局模板
- 图标优化：❤️/✈️/⭐ 改用 SVG icon ≥48dp，与文字同行

#### 闹钟
- 去掉分类：取消作息/工作/其他/自定义，按时间排序
- 显式删除：左滑显示删除按钮
- 时钟模式重做：表盘标注闹钟时间 + 下方精简列表（含开关 toggle）+ 支持删除
- 切换按钮：列表图标 ↔ 时钟图标（替换三点图标）

#### 新闻
- 内容增强：AI 摘要 150-300 字 + RSS 图片提取
- 有图/无图双容器：有图→图文并排，无图→纯文字大标题
- 新闻流：纵向滚动浏览 ≥8 条（替换固定 4 条）
- 下拉刷新：顶部下拉触发 refresh

### 六、颜色参数推送一致性

**全链路颜色参数规范**：

```
手机端 ConfigPanel
  → params.primary_color = "#FF6B35"
    → POST /api/generate (params 含 primary_color)
      → POST /api/sync (widget_data 含完整 params)
        → 车端 renderWidget() → __WIDGET_PARAMS__.primary_color
          → computePalette(primary_color) → CSS 变量
```

每个环节必须透传 `primary_color`，禁止任何环节丢弃或覆盖。

### 七、全局 Icon 系统

- 所有操作按钮从 emoji（▶ ⏸ ❤️ ✈️）替换为 SVG icon
- Icon 目录：`src/widget-templates/shared/icons/`
- 尺寸标准：操作按钮 ≥48dp，装饰图标 ≥32dp
- 引用方式：`<svg class="icon icon-{name}">` + CSS sizing

## 一、Demo 产品亮点与用户故事

### 1.1 一句话

**"告诉我你想要什么，AI 帮你生成一张专属的车机桌面卡片。"**

### 1.2 核心亮点

| # | 亮点 | 打动点 |
|---|------|--------|
| 1 | **秒级生成** | 用户填完参数，≤5s 看到一张可运行的车机卡片，现场感知 AI 的能力 |
| 2 | **所见即所得** | 手机端预览严格模拟车机 1/3 屏卡片（896×1464），确认后秒级同步到车 |
| 3 | **精致如原生** | 视觉品质与车机系统应用一致——精致、简洁、风格匹配，不是"能跑就行"的毛坯 |
| 4 | **个性化表达** | 同一个"纪念日"组件，每个人的主题、配色、文案都不一样——我的车桌面就是我的作品 |

### 1.3 v0.1 组件类型总览

v0.1 包含 **2 大类、共 4 种组件**：

| 类别 | 组件 | 说明 |
|------|------|------|

| 🔧 实用类 | **闹钟** | 快捷设置闹钟 + 下一个闹钟时间展示 |

#### 纪念日组件的 3 个主题

纪念日是一个统一组件，支持"倒数"和"正数"两种模式。通过不同的主题预设覆盖多种场景：

| 主题 | 模式 | 典型场景 | 情感内核 |
|------|------|---------|---------|
| 💕 恋爱纪念 | 正数（在一起第X天） | 恋爱、结婚纪念日 | 甜蜜、温暖 |
| 👶 宝宝成长 | 正数（出生第X天） | 宝宝出生、重要里程碑 | 珍惜、成长 |
| 🏖️ 放假倒计时 | 倒数（距离X还有X天） | 国庆、春节、年假 | 期待、活力 |

### 1.4 Demo 演示用户故事（建议准备 3 个）

#### 故事 A：恋爱纪念日（情感共鸣 → 自传播）

> 小王和女朋友的三周年纪念日快到了。他打开手机上的「AI小组件」，选择"纪念日"模板下的"恋爱纪念"主题，输入日期和标题"在一起的第___天"，AI 自动生成了一句副标题"每一天都算数"，他选了一个温暖的渐变风格。不到5秒，一张精美的纪念日卡片生成了。他点击"同步到车机"，下次上车时，行车桌面上就出现了这张卡片，每天数字都在跳动。女朋友坐上副驾看到，截图发了朋友圈。

**打动管理层的点**：自传播。用户自发截图分享 → 品牌曝光 → 潜在购车转化。

#### 故事 B：放假倒计时（高频共鸣 → 社交话题）

> 打工人小赵距离国庆假期还有 47 天，他用"纪念日"模板的"放假倒计时"主题，创建了一个国庆倒计时。AI 自动生成了副标题"假期在向你招手 🏖️"，配了一个明亮活力的风格。每天上车通勤，那个不断减小的数字就是最大的精神支柱。同事看到了也想搞一个——"你这个怎么弄的？"

**打动管理层的点**：天然社交话题。每逢假期前都是传播高峰，可运营节日主题活动。

#### 故事 C：宝宝成长记录（情感 + UGC 飞轮）

> 新手爸爸老张给刚出生的宝宝创建了一个"宝宝出生第___天"的成长卡片，配了一个柔和的主题风格。每天上车看到数字增长，是一种小确幸。车友群里有人问"你这个桌面卡片怎么弄的"，飞轮开始转。

**打动管理层的点**：UGC 飞轮雏形。用户好奇 → 下载使用 → 自己创建 → 再传播。

### 1.5 Demo 演示脚本建议

整个演示控制在 **3 分钟以内**，节奏如下：

1. **30s 讲痛点**："车载商店应用少，开发者不愿来，但用户有大量长尾个性化需求"
2. **60s 现场演示**：打开手机 → 首页 8 个场景图标一目了然 → 点击"实时天气" → 配置面板选风格/配色 → "生成卡片" → 4步AI创作动效 → 预览页卡片渲染完成 → "同步到车机" → 成功动画

4. **60s 讲故事**：小米车主群体特征（年轻、科技、分享欲） → UGC 飞轮 → 商业可能性（模板市场、节日主题运营）

## 二、关键体验打磨点 & 技术 PoC 重点

### 2.1 必须打磨的体验（Demo 成败取决于此）

#### ⭐ P0: 视觉品质——精致、简洁、与车机风格匹配

这是 Demo 的"第一印象"，必须投入最多精力打磨。

**核心审美标准**：
- **精致**：细节经得起推敲——字间距、色彩过渡、元素对齐、圆角一致性
- **简洁**：1/3 屏卡片上信息不超过 3 层（主数字 + 标题 + 副标题），留白充分
- **与车机匹配**：色调、字体、圆角、阴影等视觉语言与车机系统 UI 一脉相承，不突兀

**具体要求**：
- 字体层级清晰：主数字（天数/倒计时）极大且醒目，副标题适中，辅助信息最小
- 配色和谐：每种预设风格都是设计师调过的完整色板，不允许 AI 自由选色
- 暗色模式优先：车机绝大多数时间处于暗色主题，所有组件必须在暗色下优先适配
- 动效克制但有品质感：数字跳动、渐入等微动效，要精致不要花哨
- 严格适配 896×1464 卡片尺寸：竖向布局，内容居中，上下留白
- 每个组件的每种风格，在不同内容长度下都不能"破版"（标题过长时截断/缩字）

**实现策略**：
- 不依赖 AI 从零生成 CSS，AI 只选择预定义的「风格包」+ 填充用户参数
- 每种组件类型 × 每种风格 = 一个精心设计的 H5 模板，AI 做参数注入
- Design Tokens 强制注入且不可覆盖，保证视觉底线
- **设计师必须逐一审核每个模板×风格的渲染效果**，签字确认后才纳入

#### ⭐ P0: 生成速度

**目标**：从用户点击"生成"到看到预览 ≤ 5 秒。

**拆解（L1 模板参数化方案）**：
- AI 推理（理解意图 + 输出 JSON 参数）：≤ 2s
- 前端模板引擎渲染 H5：≤ 1s
- 预览加载：≤ 1s
- 预留 buffer：1s

L1 方案下 AI 不写代码，只输出 JSON，所以生成速度很快，5s 是保守估计。

#### P1: 手机端预览的真实感

**目标**：手机端预览 ≈ 车机实际效果，减少"到车上一看不一样"的落差。

**具体要求**：
- 预览区域严格按 896×1464 比例渲染（260×434 等比缩放卡片）
- 使用和车机完全一致的 Design Tokens 和字体
- 浅色背景 + 圆角阴影卡片，聚焦内容本身
- AI 摘要条展示生成结果描述，AI 洞察条展示模板能力亮点
- 预览页通过 iframe srcdoc 加载真实 H5 模板，确保所见即所得

#### P1: 同步到车的"仪式感"

**目标**：用户点击同步后，有明确的成功反馈。

**具体要求**：
- 手机端：同步动画 + "已同步到你的车" 提示
- 车端：新组件首次出现时有微妙的入场动画（淡入即可）
- 如果车机不在线，手机端提示"已保存，下次启动车辆时自动同步"

### 2.2 技术 PoC 验证重点

| # | PoC 项目 | 验证目标 | 成功标准 | 失败的后备方案 |
|---|---------|---------|---------|--------------|
| 1 | AI 参数生成可用性 | AI 能否准确理解用户意图并输出合法 JSON 参数 | JSON 合法率 ≥ 99%，意图理解准确率 ≥ 95% | 简化 AI 角色为纯文案生成，参数全由用户手填 |
| 2 | 车端 WebView 渲染性能 | 896×1464 WebView 加载和渲染 H5 组件的性能 | 首屏渲染 ≤ 2s，内存占用 ≤ 50MB，60fps | 精简 H5 复杂度；预渲染为静态图片 |
| 3 | 跨端同步链路 | 手机创建 → 云端 → 车端推送的全链路时效 | 车机在线时端到端 ≤ 5s | 降级为手动刷新/轮询 |
| 4 | 每日新闻数据源 | 新闻 API 的稳定性和内容质量 | 每日可获取 ≥ 5 条有效新闻摘要 | 退化为 RSS 聚合或人工运营 |

##### AI 在 L1 中的价值

虽然 L1 看起来 AI 只输出了 JSON，但 AI 在以下场景中的价值不可替代：

| 场景 | AI 做了什么 | 没有 AI 的话 |
|------|-----------|-------------|
| 副标题为空 | AI 根据主题和日期生成有创意的文案 | 用户必须自己想文案 |
| v0.5 自然语言输入 | "帮我做一个国庆倒计时" → AI 自动选模板、选风格、填参数 | 用户必须手动选择每一项 |
| v0.5 智能推荐 | 临近春节时 AI 主动推荐"要不要做个春节倒计时？" | 纯被动等用户操作 |

##### 三级生成策略演进

| 级别 | 方式 | AI 输出 | 适用阶段 |
|------|------|--------|---------|

| L2 模板+微调 | AI 基于模板骨架生成局部 CSS 变体 | JSON + CSS 片段 | v0.5 |
| L3 自由生成 | AI 完整生成 H5 代码 | HTML+CSS+JS | v1.0+ |

#### 难点 2：车端 WebView 性能与稳定性

**问题**：车机 Android WebView 版本可能较低、内存受限、需长时间运行不崩溃。

**卡片尺寸确认**：896×1464 逻辑像素（1/3 屏）。

**建议**：
- 确认车机 WebView 内核版本（Chromium 版本），决定可用的 CSS/JS 特性集
- H5 组件必须极度轻量：单组件总资源 ≤ 100KB（含 HTML+CSS+JS+图片）
- 禁止使用重型框架（React/Vue），用 Vanilla JS + CSS 实现
- 所有动效用 CSS Animation / requestAnimationFrame，禁用 JS 定时器密集更新
- 内存泄漏防护：组件生命周期管理，不可见时暂停所有定时器和动画
- WebView 崩溃隔离：宿主 App 捕获 WebView 崩溃，自动重载而非影响桌面

**建议做一次基准测试**：
```
测试项: 在目标车机上运行一个包含 CSS 动画的 H5 页面（896×1464 WebView）24 小时
观测指标: 内存趋势、帧率、WebView 崩溃次数
目标: 内存无持续增长，帧率稳定 60fps，0 崩溃
```

#### 难点 3：跨端同步

**问题**：手机创建后如何快速、可靠地将组件推送到车机。

**建议方案**：

```
手机端                    云端                      车端
  │                        │                        │
  │── 1.上传组件产物 ──────►│                        │
  │   (H5 zip + metadata)  │                        │
  │                        │── 2.推送通知 ──────────►│
  │                        │   (MQTT/长连接)          │
  │                        │                        │── 3.拉取组件产物
  │                        │◄───────────────────────│
  │                        │── 4.返回 H5 zip ──────►│
  │                        │                        │── 5.解压 + 加载
  │◄── 6.同步确认 ─────────│◄── 确认完成 ───────────│
```

**关键设计决策**：
- 组件产物以 zip 包形式存储（HTML + CSS + JS + 图片资源）
- 云端存储用对象存储（如 S3/MinIO），元数据用 KV/MySQL
- 车端组件缓存在本地，支持离线运行
- 车机首次启动时全量同步，之后增量更新
- 如果车机团队已有推送通道（MQTT/自有长连接），优先复用，不新建

**Demo 阶段简化方案**：
如果推送通道搭建成本高，Demo 可以用轮询替代（车端每 5s 检查一次有无新组件），体验可接受且大幅降低开发成本。

#### 难点 5：每日新闻数据源

**问题**：新闻组件需要每日更新的内容源。

**建议方案**：
- 云端新闻聚合服务，每日定时拉取 → AI 摘要 → 缓存
- 数据源优先级：官方新闻 API（如头条/腾讯新闻开放平台）> RSS 聚合 > 人工运营
- 车端组件每次展示时从云端拉取最新缓存（非实时请求新闻源）
- 分类支持：综合、科技、汽车（v0.1 只做综合即可）

**Demo 简化方案**：
如果新闻 API 接入周期长，Demo 可以用预置的模拟数据（每天手动更新几条），核心是验证卡片上新闻的展示体验。

#### 难点 6：行车桌面卡片内多组件切换

**问题**：1/3 屏卡片只能展示一个组件，但用户可能有多个。

**v0.1 方案：手动点击切换**

不做智能排序，采用最简单可靠的方案：
- 卡片底部显示圆点指示器（· · ● · ·），标示当前是第几个组件
- 卡片上方左右各一个轻量的切换箭头按钮（← →），点击切换上一个/下一个
- 组件顺序 = 用户创建顺序（最新的在最前面），支持在手机端拖拽排序
- 用户上次查看到第几个组件，下次上车自动恢复到同一个

**交互细节**：
- 切换箭头默认半透明，触摸时高亮，3秒无操作后淡出（减少视觉干扰）
- 切换动画：简单的左右滑动过渡，200ms duration
- 如果只有 1 个组件，不显示箭头和圆点

### 3.3 技术选型总结

| 模块 | 推荐选型 | 备选 | 选型理由 |
|------|---------|------|---------|
| AI 生成 | LLM API (GPT-4o / Claude) + JSON Mode | 自建微调模型 | Demo 阶段追求速度和灵活性，不值得训练模型 |
| AI 服务框架 | Python FastAPI | Node.js | Python 生态对 LLM 集成更成熟 |
| H5 模板引擎 | Vanilla JS + Handlebars/Mustache | Lit/Preact | 最轻量，兼容性最好，车机 WebView 友好 |
| 手机端 | H5 Web App（最快交付） | React Native | Demo 阶段手机端不是重点，追求最快上线 |
| 云端存储 | 对象存储 + MySQL/Redis | MongoDB | 组件产物用对象存储，元数据结构简单用关系型 |
| 新闻数据源 | 新闻 API + AI 摘要 + 缓存 | RSS / 人工运营 | Demo 可先用模拟数据，后续接入真实源 |
| 推送同步 | MQTT（如已有通道）/ 轮询（Demo降级） | WebSocket | 复用车机已有推送通道成本最低 |
| 车端宿主 | Android App + WebView | 小程序容器 | WebView 是最直接的方案，后续可升级 |

## 精品模板核心功能和交互

> 本章详细定义每个精品模板的核心功能点、交互流程和 UX 设计标准。
> 所有模板遵循统一的 **Liquid Glass** 设计语言，暗色优先，896×1464 逻辑像素。

### 一、天气卡片 (Weather)

**核心功能**：
- 实时温度、天气状况（晴/多云/雨等）、体感温度
- 三项指标条：湿度、风力、紫外线（毛玻璃胶囊）
- 3 日预报：明天/后天/周末，高低温
- AI 建议条：基于天气给出穿搭/出行建议（如"天气晴好，适合户外活动。紫外线较强，建议涂抹防晒霜。"）
- 动态粒子背景：根据天气类型切换（晴天光斑、雨天水滴、雪天飘雪）

**交互功能**：
- **切换地点**：点击左上角城市名 → 弹出卡片内浮层（半透明遮罩 + 深色圆角面板）→ 搜索栏 + 常用城市列表 → 点击城市切换 → 面板收起，数据刷新
- 浮层样式：720×960 深色面板，cornerRadius:28，居中定位，搜索栏置顶，城市列表垂直排列（城市名 + 所属省份/国家）

**参考**：Apple Weather 简洁布局 + 动态背景，Google Weather 的指标条展示方式

---

### 二、音乐播放器 (Music)

**核心功能**：
- 大尺寸专辑封面（AI 生成或原始封面，毛玻璃边框）
- 歌曲标题 + 艺术家名
- 滚动歌词预览（当前行高亮）
- 频谱可视化动画条
- 进度条：已播放时间 / 总时长
- 播放控制：上一首 / 播放暂停 / 下一首

**交互功能**：
- **无内容兜底态**：当没有音乐在播放时展示兜底 UI —— 居中音符图标 + "还没有在播放的音乐" 提示 + "去选择音乐" 按钮 + 底部波纹装饰
- 播放/暂停：点击播放按钮切换状态，频谱动画同步启停
- 切歌：点击上一首/下一首，封面和信息刷新（附淡入过渡）

**数据来源**：通过 JSBridge MediaSession 接口获取车端当前播放媒体信息

**参考**：Apple Music 的大封面 + 简洁控制，Spotify 的播放条

### 三、日历日程 (Calendar)

**核心功能**：
- 大号日期数字（当天日期）+ 月份/年份 + 农历信息
- 星期标签（右上角）
- 时间线事件列表：每个事件包含时间（HH:MM）、彩色圆点（类别标记）、事件标题、地点/备注
- 下一个会议倒计时提示条："距下次会议还有 XX 分钟"

**交互功能**：
- **新增日程**：点击"+"按钮 → 弹出卡片内浮层（白色圆角面板，light theme）→ 包含以下字段：
  - 标题输入（如"产品评审会议"）
  - 地点输入（如"3楼会议室 A"）
  - 全天开关（toggle）
  - 开始/结束时间选择（蓝色文本显示日期时间）
  - 提醒设置（如"15 分钟前"）
  - 颜色标记选择（5 色圆点：蓝/绿/橙/红/紫，当前选中带外框）
  - 备注输入
  - 顶部"取消"/"保存"操作按钮
- 浮层样式：720×1060 白色面板，cornerRadius:28，居中定位

**参考**：Apple Calendar 的时间线布局，Google Calendar 的新建事件流程

### 四、每日新闻 (News)

**核心功能**：
- 顶部标题 "每日简报" + 日期
- 头条新闻区：配图 + 分类标签（科技/汽车/财经/体育等）+ 标题 + 来源和时间
- 次要新闻列表：每条包含分类标签 + 标题 + 来源/时间（无配图，紧凑排列）
- 分类标签颜色体系：科技(蓝)、汽车(绿)、财经(金)、体育(红)

**交互功能**：
- **查看全文**：点击任意新闻条目 → 跳转全文阅读页（独立长页面，可滚动）
- 全文页结构：
  - 顶部导航栏（返回箭头 + "新闻详情" + 分享按钮）
  - 题图（全宽，480px 高）
  - 分类标签胶囊（圆角背景色块）
  - 大号标题（32px，加粗）
  - 元信息行（来源 · 发布时间 · 预计阅读时长）
  - 正文段落（18px，1.8 行高，#FFFFFFCC）
  - 小标题分隔（24px，加粗）
  - 底部互动栏（点赞数 · 评论数 · 收藏）
- 全文页为深色背景（#0A0E14），沉浸式阅读体验

**数据来源**：RSS 聚合 + AI 摘要生成（30 分钟缓存）

**参考**：Apple News 的卡片式布局，Google News 的分类体系

### 五、闹钟 (Alarm)

**核心功能**：
- 闹钟开关（左上角 toggle，绿色=开启）
- 距响铃倒计时文案（如"16 小时 36 分钟后响铃"）
- 大号时间显示（08:00）+ AM/PM 标记
- 重复日设置摘要（如"周一至周五"）
- 下一个闹钟预览条（毛玻璃面板：时间 · 重复规则 · 标签）

**交互功能**：
- **开关闹钟**：直接点击 toggle 切换启用/禁用，toggle 状态动画（绿→灰）
- **新增闹钟**：点击"+"按钮 → 弹出卡片内浮层（深色圆角面板 #1A1F2E）→ 包含以下字段：
  - 时间选择器：大号数字（小时:分钟），背景色块选择区 + AM/PM 切换按钮
  - 重复日选择：7 天圆形按钮（一~日），选中为蓝色高亮，未选为暗色
  - 标签设置行（如"上班闹钟"）
  - 振动开关（toggle）
  - 顶部"取消"/"保存"操作按钮
- 浮层样式：720×1000 深色面板，cornerRadius:28，居中定位

**参考**：iOS 原生闹钟的简洁性，去掉铃声/贪睡等非核心功能

### 六、纪念日 — 恋爱主题 (Love)

**核心功能**：
- 全屏背景照片（情侣主题）
- 毛玻璃信息面板（底部）：
  - 超大号数字（如 365）
  - 描述文案（如"天的相伴"）
  - 副标题（如"与你相伴的每一天"）
  - 日期区间（如 2025.03.22 — 2026.03.22）
- 心形粒子系统动画

**设计要点**：
- 毛玻璃面板 opacity 25%（#FFFFFF40），确保文字可读性
- 暗化叠加层 50%（#1A0A1580），让文字在照片上可辨识
- 主数字 #FFFFFFD0（高对比），副文字 #FFFFFFA0（次级层次）
- 面板需在 Safety Margin（48px）内

**参考**：iOS 锁屏小组件的照片+信息叠加模式

### 通用交互规范

**卡片内浮层（Popup）**：
- 触发方式：点击卡片内的功能入口（如城市名、"+"按钮）
- 视觉：半透明遮罩（#00000080）+ 居中圆角面板（cornerRadius:28）
- 深色模板用深色面板（#1A1F2E），浅色模板用白色面板（#FFFFFF）
- 面板宽度固定 720px（卡片 896px - 两侧 88px 边距），高度按内容自适应
- 关闭方式：点击遮罩区域 / 点击"取消"按钮
- 动画：面板从底部滑入，遮罩 fade in（200ms ease-out）

**兜底态设计**：
- 所有卡片在无数据时需有友好兜底 UI（如音乐无播放、日程无事件、新闻加载中）
- 兜底 UI 包含：图标/插画 + 提示文案 + 操作按钮（如有）
- 视觉保持与正常态一致的设计语言，不突兀

**数据刷新**：
- 天气/新闻：30 分钟缓存，过期后静默刷新
- 日程/闹钟：本地数据优先，JSBridge 同步
- 音乐：实时跟随车端播放状态

**可读性标准（倒计时卡片通用）**：
- 照片背景上的文字必须有毛玻璃面板承托
- 毛玻璃面板 opacity ≥ 25%（不低于 #FFFFFF40）
- 暗化叠加层 opacity ≥ 50%
- 主数字 opacity ≥ 80%（#FFFFFFD0），副文字 opacity ≥ 62%（#FFFFFFA0）
- Safety Margin: 内容距卡片边缘 ≥ 48px

### 阶段性成果总结（v0.4 完成时）

**产品形态**：
- 9 个精品模板，每个都有独立的卡片内交互
- 3 条生成链路：模板配置(<2s) / 自然语言微调(2-8s) / AI 编程(15-30s)
- 手机→云端→车机全链路闭环（Android APK 真机验证）

**技术指标**：
- 每模板 ≤ 100KB，无重型前端框架
- 模板配置生成 < 2 秒（纯客户端，不调 LLM）
- Mock 优先渲染策略，首屏无空白等待
- 8 小时连续运行无崩溃（车端 WebView 稳定性）

**设计体系**：
- 三层配色架构（Clean/Mood/Photo），12 色色板
- Color Engine v2 双模式，任意颜色→可区分的品质渐变
- 15 张 AI 生成预设背景图
- card.pen 设计稿 100% 对齐

**数据能力**：
- 天气：和风天气 API + 30 分钟缓存
- 新闻：12 个 RSS 源 + 图片提取 + 500 字摘要
- 日程：动态模拟 + localStorage 持久化
- 音乐：JSBridge MediaSession 桥接
